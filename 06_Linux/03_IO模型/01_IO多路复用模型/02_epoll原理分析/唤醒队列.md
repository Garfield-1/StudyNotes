总览：谁睡、谁醒、中间靠什么

ep_poll()（epoll_wait 路径）在没事件时，把 current（当前要等的线程）以 wait_queue_entry 的形式挂到 ep->wq 上，然后 schedule_hrtimeout_range() 进入睡眠（TASK_INTERRUPTIBLE）。

你在 epoll_ctl(ADD) 时，内核会在 ep_insert → ep_item_poll → ep_ptable_queue_proc 里，给被监视的 socket file 的 poll 等待队列（wait_queue_head_t *whead）挂上 ep_poll_callback。

socket 变活跃（数据到达、可写等）时，协议栈 / 驱动会在 socket 自己的等待队列上做 wake_up*() → 内核通用 waitqueue 机制会调用你注册的那个 wait.func，也就是 ep_poll_callback()。

ep_poll_callback() 里把对应的 epitem 放进 ep->rdllist（或 ovflist），并对 ep->wq 执行 wake_up(&ep->wq) → 之前在 ep_poll() 里睡在 ep->wq 上的 epoll_wait 线程被唤醒。

下面按代码把“挂钩子”和“唤醒”两点钉死。



1) ep_poll() 把谁放到 ep->wq 上睡

ep_poll() 在需要阻塞时：



init_waitqueue_entry(&wait, current)

__add_wait_queue_exclusive(&ep->wq, &wait)：把 current 加到 ep->wq

随后在循环里 schedule_hrtimeout_range(...) 真正让出 CPU

也就是说：等在 epoll 实例上的线程，等的是 ep->wq，不是直接睡在 socket 的 sk_sleep 上（那是下一步的回调链）。



2) socket 活跃时，为什么会进到 ep_poll_callback()

在 ep_ptable_queue_proc()（poll_table 的 _qproc）里，内核为每个 epitem 分配 eppoll_entry，并把等待队列项初始化成 带函数的 waitqueue entry：



```c
eventpoll.c

Lines 1322-1329

​	if (epi->nwait >= 0 && (pwq = kmem_cache_alloc(pwq_cache, GFP_KERNEL))) {

​		init_waitqueue_func_entry(&pwq->wait, ep_poll_callback);

​		pwq->whead = whead;

​		pwq->base = epi;

​		if (epi->event.events & EPOLLEXCLUSIVE)

​			add_wait_queue_exclusive(whead, &pwq->wait);

​		else

​			add_wait_queue(whead, &pwq->wait);
```



含义：



whead：目标 file->poll() 暴露出来的 socket（或设备）侧等待队列头。

pwq->wait 的回调 = ep_poll_callback：以后 socket 那一侧 wake_up(whead) 时，就会 间接调用 ep_poll_callback()（并传入 key，里面携带 pollflags（就绪事件位） 等信息，具体取决于唤醒路径）。

所以：socket “活跃”在实现上通常表现为：内核在 socket 的 waitqueue 上 wake_up*，从而触发 ep_poll_callback。



3) ep_poll_callback() 怎么唤醒 ep_poll() 睡着的进程

核心片段：



```c
eventpoll.c

Lines 1216-1281

​	read_lock_irqsave(&ep->lock, flags);

​	// ... EPOLLONESHOT / pollflags 过滤 ...

​	if (READ_ONCE(ep->ovflist) != EP_UNACTIVE_PTR) {

​		// ep_scan_ready_list / ep_send_events 正在“偷 rdllist”时：先链到 ovflist

​		// ...

​		goto out_unlock;

​	}

​	if (!ep_is_linked(epi) &&

​	    list_add_tail_lockless(&epi->rdllink, &ep->rdllist)) {

​		ep_pm_stay_awake_rcu(epi);

​	}

​	if (waitqueue_active(&ep->wq)) {

​		// EPOLLEXCLUSIVE 时 ewake 逻辑略（影响返回值/唤醒传播细节）

​		wake_up(&ep->wq);

​	}

​	if (waitqueue_active(&ep->poll_wait))

​		pwake++;
```



对你这个问题最关键的两句是：



list_add_tail_lockless(..., &ep->rdllink)：把该 epitem 标记进 epoll 的就绪链表（或走上面的 ovflist 分支，等扫描结束再并回 rdllist，避免丢事件）。

wake_up(&ep->wq)：如果 ep->wq 上确实有人（ep_poll() 挂的 waiter），就把它们唤醒。

被唤醒的 ep_poll() 线程醒来后会继续循环检查 ep_events_available()，发现 eavail 为真，最终走到 ep_send_events() 把事件 copy_to_user。



另外：ep->poll_wait 那条 ep_poll_safewake(&ep->poll_wait) 是为了 “epoll 套 epoll / 上层也在 poll 这个 epoll fd” 的场景，在 锁外 做更谨慎的唤醒；你问的 ep_poll() 等 socket 的主线通常是 wake_up(&ep->wq)。



4) 用一句话串起来

ep_poll() 睡在 ep->wq；socket 就绪时在 socket 的 waitqueue 上 wake_up，触发挂在 socket 上的 ep_poll_callback；它把 epitem 入 rdllist（或 ovflist）并 wake_up(&ep->wq)，从而唤醒 ep_poll()。



5) 两个常见误区（帮助你自检理解）

不是 ep_poll() 直接睡在 socket 队列上：它睡在 ep->wq；socket 队列上挂的是 ep_poll_callback 这种 “桥接回调”。

ep_send_events/ep_scan_ready_ready_list 期间：ep->ovflist 非 EP_UNACTIVE_PTR 时，回调会把事件先堆 ovflist，避免与 “无锁扫描/拷贝用户内存” 冲突；扫描结束再并回 rdllist，ep_poll() 仍可能通过后续 wake_up / ep_events_available 看到事件。

如果你想把 TCP 收包路径里具体是哪一个 wake_up 触达 socket 的 whead，那是 网络子系统（sock） 的另一条调用链；但就 epoll 机制本身而言，边界就是：socket waitqueue → ep_poll_callback → ep->wq 唤醒 ep_poll()。







## 1) 用户态“epoll 监听 epoll”时，会用到 `ep->poll_wait` 吗？

会。

当用户把一个 `epoll fd`（内层）加入另一个 `epoll`（外层）时，外层在建立监听关系时会走被监控文件的 `f_op->poll()`（文件轮询回调）。
而 `eventpoll` 的 `->poll()` 实现（`ep_eventpoll_poll`）里会执行：

- `poll_wait(file, &ep->poll_wait, wait)`

这一步就是把“外层 epoll 的等待项”挂到内层的 `ep->poll_wait` 上。
所以内层一旦有 ready events（就绪事件），唤醒 `ep->poll_wait` 就能通知到外层。

------

## 2) 为什么 epoll 内部要用 `ep->poll_wait`？

因为 epoll 既是：

- 事件聚合器（event aggregator，事件聚合器）：通过 `epoll_wait` 给应用返回事件（对应 `ep->wq`）；
- 也是一个可轮询文件（pollable file，可被 poll 的文件描述符）：别人可以对它执行 `poll/select/epoll`（对应 `ep->poll_wait`）。

`ep->poll_wait` 就是第二个角色必须的“对外等待队列”。
没有它，`epoll fd` 就无法像普通 fd 一样被上层多路复用器观察，嵌套 epoll 也无法工作。

------

## 3) 从软件架构看，`ep->poll_wait` 是为谁设计的接口？

面向的是 VFS poll/select/epoll 框架（虚拟文件系统轮询框架），以及所有“把 epoll fd 当被监控对象”的上层组件。

可以理解成两层 API（应用程序编程接口）：

- `ep->wq`：面向“直接消费者”——调用 `epoll_wait()` 的线程；
- `ep->poll_wait`：面向“上层观察者”——调用 `poll()/select()` 或外层 `epoll` 的框架路径。

一句话：
`ep->poll_wait` 不是给普通 `epoll_wait` 主循环用的，而是给“观察 epoll fd 状态变化”的上层复用层用的标准接口。



## 1) 用户态“epoll 监听 epoll”时会用 `ep->wq` 吗？

会，但位置不同于 `ep->poll_wait`：

- 外层监听内层这条“建链/传播”路径主要用的是内层的 `ep->poll_wait`（通过内层 `->poll()` 挂等待项）。
- 当内层真的有事件后，最终要把事件返回给某个线程，那个线程如果在调用 `epoll_wait()`（不管是内层还是外层 epoll fd），它睡眠/唤醒走的是对应实例的 `ep->wq`。

所以在“epoll 监听 epoll”场景里，两者通常都会出现：

- `poll_wait` 用于层间通知；
- `wq` 用于最终把事件交给 `epoll_wait` 调用者。

------

## 2) 为什么 epoll 内部要有 `ep->wq`？

因为 `epoll_wait()` 是阻塞式等待接口。
内核必须有一个等待队列让调用 `epoll_wait` 的任务睡眠，并在就绪时唤醒它们，这就是 `ep->wq` 的职责。

没有 `ep->wq`，`epoll_wait` 就无法高效阻塞等待，只能轮询（busy loop，忙等）。

------

## 3) 从软件架构看，`ep->wq` 是为谁提供接口？

面向的是 `epoll_wait` 系统调用路径（以及内核中等价的等待事件消费者）。
也就是“直接消费 epoll ready list（就绪链表）”的那一层。

可对照理解：

- `ep->wq`：给 `epoll_wait` 消费者层。
- `ep->poll_wait`：给 `poll/select/外层 epoll 观察者层`。

一句话总结：
`ep->wq` 负责“把事件交给等待 `epoll_wait` 的线程”，`ep->poll_wait` 负责“把 epoll fd 已就绪这个事实通知给更上层复用器”。
